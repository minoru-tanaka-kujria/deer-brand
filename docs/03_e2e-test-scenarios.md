## シナリオ1: 新規ユーザーのフル購入フロー

**概要:** 初めて来訪したユーザーが Google ログインして、商品を選び、決済まで完了する基本フロー

### 前提条件
- ブラウザのキャッシュ・Cookieをクリアした状態（未ログイン）
- Firebase Auth / Firestore が稼働中
- Stripe がテストモードで稼働中（STRIPE_SECRET_KEY が `sk_test_...` であること）
- テスト用 Google アカウントが使用可能であること
- ペット写真として使うJPEGファイル（任意の画像）が手元にあること

### 手順

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | `upload.html` を開く | ルックブックグリッドが表示される。ログインバナー等は表示されない |
| 2 | ルックブックから好みの商品カードをクリック | カードにテラコッタボーダーとチェックバッジが表示される |
| 3 | 「この商品を選ぶ」ボタンをクリック | Step 2（プリント配置）へ遷移。プログレスバーが Step 2 を active に |
| 4 | 配置オプションを1つクリックして選択 | 選択カードにボーダーが表示される |
| 5 | 「次へ — カラー選択」をクリック | Step 3（カラー選択）へ遷移 |
| 6 | カラースウォッチを1つ選択 | スウォッチにボーダーが表示される |
| 7 | 「次へ — サイズ選択」をクリック | Step 4（サイズ選択）へ遷移 |
| 8 | サイズボタン（例: M）をクリック | ボタンが選択状態に |
| 9 | 「次へ — 写真アップロード」をクリック | Step 5（写真アップロード）へ遷移 |
| 10 | 頭数「1」が選択された状態を確認 | デフォルトで1頭選択済み |
| 11 | アップロードゾーンをクリックしてJPEGファイルを選択 | プレビュー画像がゾーン内に表示される |
| 12 | 「アートを生成する」をクリック | Step 6（プレビュー）へ遷移。スピナーが表示される |
| 13 | 生成完了を待つ（30秒〜1分） | モックアップ画像と「この仕上がりで注文する」ボタンが表示される |
| 14 | 「この仕上がりで注文する」をクリック | Step 7（お届け先）へ遷移 |
| 15 | お届け先フォームに入力（テストデータ参照） | 全フィールドに入力完了 |
| 16 | 「次へ — お支払い」をクリック | Step 8（お支払い）へ遷移 |
| 17 | Stripe Elementsカード入力欄に `4242 4242 4242 4242` を入力 | カード番号フォームが受け付ける |
| 18 | 有効期限 `12/28`、CVC `123` を入力 | 入力完了 |
| 19 | 「注文内容を確認する」をクリック | Step 9（注文確認）へ遷移 |
| 20 | 注文内容サマリーを確認 | 選択した商品・配置・カラー・サイズ・住所が表示されている |
| 21 | 「注文を確定する」をクリック | 決済処理が開始される |
| 22 | 決済成功後の完了画面を確認 | `DEER-YYYYMMDD-XXXX` 形式の注文IDが表示される |
| 23 | Stripe ダッシュボードを確認 | PaymentIntentが `succeeded` になっている |
| 24 | Firestore `orders` コレクションを確認 | `status: "pending"` または Webhook受信後 `status: "preparing"` の注文ドキュメントが存在する |

### テストデータ

**お届け先:**
```
姓: 山田　名: テスト
メール: test-e2e@deer-test.com
電話: 090-0000-0001
郵便番号: 150-0001
都道府県: 東京都
住所1: 渋谷区神南1-2-3
住所2: （空欄）
```

**決済:**
```
カード番号: 4242 4242 4242 4242
有効期限: 12/28
CVC: 123
```

### 期待結果（最終確認）
- 完了画面に注文IDが表示されること
- Firestoreに注文ドキュメントが作成されること（`userId`・`paymentIntentId`・`status`が含まれること）
- Stripe Webhook が受信され、`status` が `preparing` に更新されること

## シナリオ2: クーポン使用フロー

**概要:** 既存ユーザーがクーポンコードを入力して割引を適用する

### 前提条件
- ユーザーがメールアドレスでログイン済み（またはテスト中はログインなしでも動作確認）
- Firestoreの `coupons/DEER2026` ドキュメントが存在、または APIのフォールバックデータが有効であること
- 注文フォームを Step 9（確認画面）まで進めた状態

### 手順

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | シナリオ1の手順 1〜19 を実行して Step 9 を開く | 確認画面が表示されている |
| 2 | `#couponCode` 入力欄に `deer2026` と入力（小文字） | 入力が大文字変換されて `DEER2026` と表示される（style: text-transform: uppercase） |
| 3 | 「適用」ボタンをクリック | `/api/validate-coupon` へ `{code: "DEER2026", subtotal: <現在の小計>}` がPOSTされる |
| 4 | 小計が ¥3,000 未満の場合 | `#couponStatus` に「このクーポンは¥3,000以上のご購入で使えます」と表示される。割引は適用されない |
| 5 | 小計が ¥3,000 以上の場合 | `#couponStatus` に成功メッセージ「Deer2026年記念クーポン：¥1,000割引」が表示される |
| 6 | `#confirmTotal` の合計金額を確認 | 小計から ¥1,000 が引かれた金額になっている |
| 7 | 「注文を確定する」をクリック | 割引後の金額で決済が実行される |

### テストデータ

| コード | 期待する割引 | 最低購入額 | テスト確認事項 |
|---|---|---|---|
| `DEER2026` | ¥1,000 固定 | ¥3,000 | 小計3000円以上で-1000円 |
| `WELCOME500` | ¥500 固定 | ¥0 | 金額問わず適用 |
| `SUMMER10` | 10% | ¥5,000 | 小計5000円で-500円 |
| `GIFT500` | ¥500 固定 | ¥2,000 | 小計2000円以上で-500円 |
| `INVALID999` | — | — | 無効コードのエラー表示 |
| `DEER2026`（小計¥2,999） | — | ¥3,000 | 最低金額エラー |
| 空欄 | — | — | 適用ボタン押しても何も起きないか適切エラー |

### 期待結果
- 有効コードかつ条件を満たす場合: 合計金額に割引が反映されること
- 無効コード: エラーメッセージが `#couponStatus` に表示され、合計金額は変わらないこと
- 割引後の金額で Stripe PaymentIntent が作成されること

## シナリオ3: Instagramフォロー割引フロー

**概要:** Instagramをフォローしてスクリーンショットを提出し、¥500 OFF を適用する

### 前提条件
- `OPENROUTER_API_KEY` 環境変数が設定されていること（Gemini Vision API経由）
- 注文フォームが Step 9（確認画面）に到達していること
- テスト用のスクリーンショット画像を用意（後述）

### 手順

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | Step 9 の確認画面を開く | `#igDiscountBox` が表示されている |
| 2 | 「@deer_brand_official をフォロー」リンクをクリック | Instagram の公式アカウントページが新タブで開く |
| 3 | （フォロー後）Step 2 の「フォロー画面のスクリーンショットをアップロード」エリアをクリック | ファイル選択ダイアログが開く（`accept="image/*"`） |
| 4 | 「フォロー中」ボタンが見えるスクリーンショット画像を選択 | `#igPreviewImg` に選択画像のプレビューが表示される |
| 5 | `#igVerifySection` の「スクリーンショットを確認して割引を適用」ボタンが表示されていることを確認 | ボタンが `display: block` になっている |
| 6 | ボタンをクリック | `igVerifyBtn` が disabled になり、`/api/verify-ig-follow` へ POST（base64データ） |
| 7 | `#igStatus` に「確認中...」等の状態表示を確認 | ローディング状態 |
| 8 | APIレスポンス受信後の状態を確認 | — |
| 8a | 検証成功の場合 | `#igDiscountBox` に緑ボーダー・緑背景のstyle（`.applied` クラス）が付与。`#igStatus` に成功メッセージ。合計から¥500が引かれる |
| 8b | 検証失敗（Instagram画面ではない）の場合 | `#igStatus` に「Instagramの画面が確認できませんでした」と表示 |
| 8c | 検証失敗（フォロー中でない）の場合 | `#igStatus` に「「フォロー中」の状態が確認できませんでした」と表示 |
| 9 | 割引適用後、「注文を確定する」をクリック | ¥500 引いた金額で決済実行 |

### テスト用スクリーンショット画像の準備

| テスト区分 | 使用画像 | 期待結果 |
|---|---|---|
| 正常系 | InstagramでDeer公式アカウントを実際にフォローし「フォロー中」が表示されているスクリーンショット | `verified: true`、¥500割引適用 |
| 異常系1 | 任意のWebページのスクリーンショット（Instagram以外） | `verified: false`、「Instagramの画面が確認できませんでした」 |
| 異常系2 | Instagram画面だが「フォロー」（未フォロー）状態のスクリーンショット | `verified: false`、「フォロー中の状態が確認できませんでした」 |
| エラー系 | 非常に小さい・破損した画像 | APIが `500` または `false` を返す |

### 期待結果
- 正常系: `igDiscount: true` が注文データに含まれて保存されること
- 異常系: 割引なしで注文が続行できること（ユーザーが強制されないこと）
- confidence 値が 0〜1 の範囲で返却されること

## シナリオ4: 未認証ユーザーの行動

**概要:** ログインしていないユーザーが各画面にアクセスしたときの挙動を確認する

### 前提条件
- ブラウザで未ログイン状態（または認証トークンなし）

### 手順と期待結果

#### 4-1: 未認証でマイページにアクセス

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | `account.html` を開く | ローディングスピナーが一時表示される |
| 2 | 認証状態チェック完了後 | 「ログインが必要です」の見出しが表示される |
| 3 | 「ログイン / 新規登録」ボタンが表示されていることを確認 | ボタンが押下可能 |
| 4 | ボタンをクリック | 認証モーダル（`#authModalOverlay`）が開く |

#### 4-2: 未認証で注文フォームにアクセス

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | `upload.html` を開く | Step 1（ルックブック）が表示される（認証チェックなしで操作可能） |
| 2 | Step 1〜5 を操作して Step 9（確認画面）まで進む | 認証なしでもフォームの閲覧・選択は可能 |
| 3 | 「注文を確定する」をクリック | 認証要求のモーダルが表示されるか、または処理が止まる（`userId` が null のためAPIが 400 を返す） |

#### 4-3: 未認証で `/api/get-user` に直接アクセス

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | `GET /api/get-user?uid=testuid` を Authorization ヘッダーなしで送信 | HTTP 401、`{ error: "認証トークンがありません" }` |

| 3 | 他のユーザーのUIDを指定して送信 | HTTP 403、`{ error: "他のユーザーの情報へのアクセスは禁止されています" }` |

### 期待結果まとめ
- マイページは未認証ユーザーを適切にブロックし、ログイン誘導すること
- 注文フォームは商品選択まではアクセス可能だが、注文確定時に認証が必要であること
- APIは適切な認証チェックを行い、不正アクセスを拒否すること

## シナリオ5: マイページ確認フロー

**概要:** 購入完了後のユーザーがマイページで注文情報を確認する

### 前提条件
- シナリオ1を完了して注文が作成済みであること
- ユーザーが同じアカウントでログイン済みであること

### 手順

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | `account.html` を開く | ローディング後、マイページが表示される |
| 2 | 「アカウント情報」カードを確認 | ログインしたユーザーの表示名・メールが表示されている |
| 3 | 「注文履歴」カードを確認 | シナリオ1で作成した注文が表示されている（orderId, 商品名, 金額） |
| 4 | 注文の `status` バッジを確認 | Webhook受信済みなら「制作中」（黄色）、未受信なら `pending`（表示なし） |
| 5 | 「お届け先住所」カードを確認 | シナリオ1で入力した住所が自動保存されているか確認（Firestore `savedAddresses` による） |
| 6 | 「保有クーポン」カードを確認 | シナリオ2でクーポンを使用した場合、`availableCoupons` から消えているか確認（使用済みチェックはAPI側で行う） |
| 7 | 「表示名」フィールドを変更して「プロフィールを保存」をクリック | 「プロフィールを保存しました」トーストが表示される |
| 8 | ページをリロードして表示名を確認 | 変更した表示名が保持されていること |
| 9 | 「ログアウト」をクリック | サインアウト後、`index.html` へリダイレクト |
| 10 | `account.html` を再度開く | 「ログインが必要です」画面が表示される |

### テストデータ

| 項目 | 値 |
|---|---|
| 更新後の表示名 | `テスト 太郎 更新済` |
| 確認するorderId形式 | `DEER-20260401-ABCD` |

### 期待結果
- 注文履歴が正しく表示されること（最大5件）
- プロフィール更新がFirestoreに永続化されること
- ログアウト後に認証状態がリセットされること
- 住所が自動保存されている場合、住所カードに表示されること
